# 想法精炼（Idea Refine）

## 概述

将原始想法精炼为清晰、可执行的概念，服务 TasteMate 的 Intake → Design 流程。精炼输出直接对接 `writing-plans` 技能和迭代文档模板。

## 何时使用

- 新功能请求描述模糊（"能不能让排序更聪明"）
- 迭代规划前需求方向不明确
- 多个实现方案需要权衡
- 用户提出"我想加个 X"但没有说清楚边界
- Intake 阶段收集到的需求相互矛盾

**不使用的情况：** 范围已明确的 bug 修复、已有完整验收标准的迭代任务、纯技术问题（去 systematic-debugging）。

## TasteMate 特定约束

在精炼过程中必须始终意识到以下项目约束：

- **不修改 Hermes 源码。** TasteMate 是外部 MCP server。如果想法涉及改变 Hermes 行为，必须重新设计为通过 MCP tool 结果间接影响。
- **当前迭代边界。** 范围过大的想法必须标注"后续迭代"部分。说"以后"不是推卸责任，是保护当前迭代的不做清单。
- **规则评分优先。** TasteMate 当前用规则评分（关键词匹配 + 标记位），不依赖 LLM。如果想法需要 LLM，必须标注为"迭代边界外"或"需专门评估"。
- **本地优先。** 所有数据存储在 `~/.tastemate/`，不依赖外部服务。

## 流程

### 阶段一：理解与发散

**目标：** 接受原始想法并打开它。

1. **重述想法**为一句"如何做到……"的问题陈述。强制明确实际要解决什么。

2. **问 3-5 个锐化问题：**
   - 具体为谁解决？TasteMate 的使用者是用了 `@taste` 的用户
   - 成功是什么样的？可观测的行为变化是什么？
   - 真正的约束是什么？（技术、时间、向后兼容性）
   - 之前做了什么？当前 profile 结构、评分公式能支撑吗？
   - 为什么是现在？跟当前迭代的关系是什么？

3. **生成 3-5 个方向变体**，使用以下透镜（按 TasteMate 场景挑选最相关的 2-3 个）：
   - **最简版本：** "不做新功能，能否通过配置/参数调优达到目的？"
   - **反向思考：** "如果用户不需要这个，他们真正需要的是什么？"
   - **数据驱动：** "现有 profile 数据能支撑这个吗？需要收集哪些新的 evidence？"
   - **约束增加：** "如果要保证不破坏已有 profile，方案会变成什么样？"
   - **后续迭代视角：** "如果这是后续迭代的一部分，最小可行的一步是什么？"

### 阶段二：评估与收敛

在用户对阶段一做出反应后，转向收敛：

1. **聚类**共鸣的方向为 2-3 个明显不同的方案。每个方案应在方法上有本质差异，而非同一主题的变体。

2. **压力测试**每个方案，对照三个标准：
   - **用户价值：** 谁受益？多大程度？是痛点还是痒点？与 `@taste` 的核心闭环有多相关？
   - **可行性：** 技术成本？最难的部分？是否与现有 profile 结构兼容？
   - **边界清晰：** 方案能否在一个迭代内交付？"不做"清单是否明确？

3. **暴露隐藏假设。** 对每个方案，明确命名：
   - 你押注为真但尚未验证的事情
   - 什么可能杀死这个方案
   - 你选择忽略什么（以及为什么当前可以忽略）

### 阶段三：锐化与产出

输出一个具体产物 — markdown 单页，为 Intake 和 Design 阶段做好准备：

```markdown
# [方案名称]

## 问题陈述
[一句话"如何做到……"框架]

## 推荐方向
[选定的方向及其理由 — 2-3 段]

## 关键假设待验证
- [ ] [假设 1 — 如何验证]
- [ ] [假设 2 — 如何验证]

## MVP 范围（当前迭代）
[验证核心假设的最小版本。什么在里面，什么在外面。]

## 不做（以及为什么）
- [事项 1] — [原因]
- [事项 2] — [原因]

## 后续迭代预留
- [事项] — 标注目标迭代，如"迭代三"

## 未决问题
- [在构建之前需要回答的问题]

## 与现有系统的交互
- [影响 profile 结构？评分公式？Hermes 插件？]
```

**"不做"列表可能是最有价值的部分。** 聚焦意味着对好想法说不。让 trade-off 显性化。

### TasteMate 项目特有的评估维度

在压力测试方案时，额外检查：

| 维度 | 问题 |
|---|---|
| 向后兼容 | 已有用户的 profile.json 能否正常工作？ |
| 评分公式影响 | 是否会改变 final_score 的含义或范围？ |
| MCP 协议 | 是否新增/修改 tool？参数签名是否兼容？ |
| Hermes 插件 | 排序结果的格式是否保持不变？ |
| 迭代边界 | MVP 范围内所有任务能否在一个迭代内完成？ |
| 测试策略 | 这个变更能否用 pytest 验证？是否需要新增测试类别？ |

## 反模式

- **生成 15+ 个方向。** 质量优先于数量。3-5 个精心考虑的方向胜过 15 个表面方向。
- **不要做 yes-machine。** 对弱想法用具体和善意的方式推回。
- **不要跳过"为谁解决"。** 每个好想法始于一个人的问题。
- **不要生成没有假设的方案。** 未经验证的假设是杀死好想法的第一原因。
- **不要过度工程化流程。** 三个阶段，每个阶段做好一件事。抵制添加步骤的冲动。
- **不要忽略代码库。** 现有架构是约束也是机会。精炼中引用具体文件。
- **不要跳过"后续迭代"标注。** 范围蔓延是迭代失败的第一原因。

## 红旗

- 跳过"为谁解决"和"成功标准"直接跳到技术方案
- 生成的方案全是同一思路的轻微变异
- 产出中没有"不做"列表
- 在项目约束内不可行的方案未被标注
- 新想法涉及修改 Hermes 源码但未标注为超出项目边界
- 方案要求 LLM 但未标注为"需专门评估"

## 验证

完成精炼后：

- [ ] 存在清晰的"如何做到……"问题陈述
- [ ] 目标用户和成功标准已定义（可观测行为变化）
- [ ] 探索了多个方向，而非只有第一个想法
- [ ] 隐藏假设已明确列出，并附验证策略
- [ ] "不做"列表使 trade-off 显性化
- [ ] "后续迭代预留"标注了超出当前迭代的内容
- [ ] 方案与项目约束（不修改 Hermes、本地 MCP、规则评分）兼容
- [ ] 输出是具体产物（markdown 单页），而非仅对话
- [ ] 用户确认了最终方向后才进入 Design 阶段
