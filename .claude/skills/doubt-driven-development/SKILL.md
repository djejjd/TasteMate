# 疑虑驱动开发（Doubt-Driven Development）

## 概述

自信的答案不等于正确的答案。长对话积累的上下文会悄悄将假设变成"事实"。疑虑驱动开发的纪律是：在非平凡决策确定之前，召唤一个偏重**推翻**而非认可的新鲜上下文审查者。

这不是 `/review`。`/review` 是对完成品的判决。这是飞行中的姿态：非平凡决策在纠偏成本还低的时候就接受交叉验证。

## 何时使用

决策是**非平凡**的，当满足以下至少一项：

- 引入或修改分支逻辑
- 跨越模块边界（tastemate/core 与 tastemate/tools 之间，或 tastemate 与 integrations 之间）
- 断言类型系统无法验证的属性（评分公式的数学正确性、feedback delta 的累积行为、向后兼容性）
- 其正确性依赖于未来读者看不到的上下文
- 爆炸半径不可逆（profile 数据迁移、评分公式变更、Hermes 插件协议变更、公开 schema 修改）

具体触发场景：

- 修改 `tastemate/core/scoring.py` 中的评分权重或公式
- 修改 `tastemate/core/feedback.py` 中的反馈处理逻辑
- 修改 `tastemate/schemas/` 中的 profile 数据格式
- 修改 `tastemate/storage/json_store.py` 的读写格式
- 修改 Hermes 插件的排序结果改写逻辑
- 声称非显而易见的事实："这个公式变更不影响已有 profile"

**不使用的情况：**

- 机械操作（重命名、格式化、文件移动）
- 遵循明确的、无歧义的用户指令
- 阅读或总结现有代码
- 单行变更且正确性显然
- 纯工具操作（运行测试、列出文件）
- 用户明确要求速度优先于验证

## 流程

复制此检查清单：

```
疑虑循环：
- [ ] 第一步：声明 — 写下了声明 + 为什么重要
- [ ] 第二步：提取 — 隔离了产物 + 契约，剥离推理过程
- [ ] 第三步：质疑 — 调用了新鲜上下文的对抗性审查者
- [ ] 第四步：调和 — 对照产物文本分类了每一条发现
- [ ] 第五步：停止 — 满足了停止条件（琐碎发现、3 轮、或用户覆盖）
```

### 第一步：声明 — 命名你所坚持的东西

用两到三行命名决策：

```
声明："新的反馈权重上限 0.12 不会导致已有 profile 的
       preference_fit 溢出 [0, 1] 截断范围。"
为什么重要：如果溢出，现有用户的偏好分数可能被错误裁剪，
         导致排序行为退化，且无告警。
```

如果写不出这么紧凑的声明，你拥有的不是决策，是感觉。先具象化再审查。

### 第二步：提取 — 最小可审查单元

新鲜上下文审查者需要**产物**和**契约**，不需要推导过程。

- 代码：diff 或函数 — 不是整个文件
- 决策：3-5 句话的提案 + 必须满足的约束
- 断言：声明 + 据称支撑它的证据

剥离你的推理过程。如果交给审查者的是结论，拿回来的也只会是对结论的验证。

### 第三步：质疑 — 召唤新鲜上下文审查者

审查者的 prompt **必须是对抗性的**。框架决定答案。

```
对抗性审查。找出这个产物的问题。
假设作者过于自信。寻找：
- 未声明的假设
- 未处理的边界情况
- 隐藏的耦合或共享状态
- 契约可能被违反的方式
- 可能被破坏的现有约定
- 意外输入下的失败模式

不要验证。不要总结。找出问题，或在彻底检查后明确声明找不到任何问题。

产物：<粘贴产物>
契约：<粘贴契约>
```

**只传产物 + 契约。不要传声明。** 把声明交给审查者会偏置它走向认可。

### 第四步：调和 — 将发现折叠回来

审查者的输出是数据，不是判决。你仍然是决策者。对照产物文本重读每一条发现，按以下**优先级顺序**分类：

1. **契约误读** — 审查者标记的问题是因为你提供的契约不清晰或不完整。先修契约，下一轮重新分类。
2. **有效 + 可行动** — 需要修改产物的真实问题。修改后重新循环。
3. **有效但可接受** — 问题是真实的，但修复成本超过接受成本。明确记录 trade-off。
4. **噪音** — 审查者因缺少上下文而误判。记录，跳过，并问自己：如果把那段上下文加入契约，是否能避免这次误判？

### 第五步：停止 — 有界循环，不递归

满足以下条件之一时停止：

- 下一轮只返回琐碎或已被考虑的发现，**或**
- 已完成 3 个循环（升级给用户，不要独自开始第四轮），**或**
- 用户明确说"ship it"

如果 3 轮后审查者仍持续发现实质性问题，产物可能还没准备好。向用户呈现 — 三轮未解决的循环是关于产物的信息，不是继续循环的理由。

### TasteMate 项目特定的高风险决策

以下变更自动触发疑虑驱动：

| 变更类型 | 为什么高风险 |
|---|---|
| 修改 `ScoringConfig` 的权重参数 | 影响所有用户的排序结果 |
| 修改 `_conservatively_update_existing_stable_preference` 的 delta 范围 | 影响偏好画像的累积行为 |
| 修改 profile.json 的顶层结构 | 破坏已有用户的画像文件 |
| 修改 `Ranker.rank()` 的返回类型 | 影响 Hermes 插件的排序消费逻辑 |
| 修改 `normalize_candidate` 的 ID 生成逻辑 | 改变候选的稳定标识，影响 feedback 追踪 |
| 新增 tool 或修改已有 tool 的参数签名 | MCP 协议层面的向后不兼容变更 |

## 与其他技能的互动

- **security-and-hardening**：安全隐患是疑虑驱动的天然输入。怀疑一个变更有安全影响时，先跑安全检查清单，再进入疑虑流程。
- **test-driven-development**：TDD 的 RED 步骤是将疑虑具体化 — 一个失败的测试就是一个证伪尝试。对于行为性声明，失败测试就是疑虑步骤。
- **systematic-debugging**：当审查者发现一个真实的失败模式时，切入调试流程定位和修复。
- **`/code-review`**：互补。code-review 是 PR 的事后判决；疑虑驱动是飞行中的逐决策审查。两者都用。

## 常见合理化借口

| 借口 | 现实 |
|---|---|
| "我很确定，跳过疑虑步骤" | 在新问题上，自信与正确性的相关性很低。确定的时刻正是盲点隐藏的时刻。 |
| "召唤审查者很贵" | 在生产中调试一个错误的提交更贵。检查是有界的，bug 不是。 |
| "审查者只会挑刺" | 只有不加范围约束才会。将 prompt 约束为"会使其在契约下失败的问题"。 |
| "我最后用 /code-review 做疑虑就行" | code-review 是最终关口。疑虑驱动在纠偏成本仍低时捕获错误方向。到 PR 时就晚了。 |
| "如果每一步都怀疑我根本 ship 不了" | 此技能仅适用于非平凡决策，不是每次按键。重读"不使用的情况"。 |
| "两个意见总是比一个好" | 当第二个有更少上下文并产生噪音时不是。调和，不要盲从。 |

## 红旗

- 对重命名或格式化变更启动新鲜上下文审查
- 将审查者输出视为权威而不重读产物文本
- 在用户未介入的情况下循环 > 3 轮
- 用"这个好吗？"而非"找问题"来提示审查者
- 在时间压力下跳过对高风险决策的疑虑
- **疑虑表演（可检查信号）：** 审查者连续 2+ 轮发现实质问题，但零条被分类为可行动。你在验证，不是质疑。停止并升级。
- 仅在提交后怀疑 — 那是 code-review，不是疑虑驱动

## 验证

应用疑虑驱动开发后：

- [ ] 每个非平凡决策在确定前都明确命名了声明
- [ ] 每个非平凡产物至少经过一次新鲜上下文审查
- [ ] 审查者收到的只有产物 + 契约 — 没有声明，没有推理过程
- [ ] 审查者的 prompt 是对抗性的（"找问题"），而非验证性的（"好吗"）
- [ ] 发现对照产物文本分类（非盲从），按优先级顺序：契约误读 / 可行动 / 可接受 / 噪音
- [ ] 满足了停止条件（琐碎发现、3 轮或用户覆盖）
